# Unknowns visual design system

Use this design system for every artifact the `unknowns` skill produces, so output looks like one consistent product instead of a different improvised style each time. This takes precedence over generic `artifact-design` guidance for anything generated by this skill — use `artifact-design` for accessibility/contrast/responsiveness fundamentals, use this file for the concrete palette, type, and components.

Warm, editorial, quiet-confidence tone: paper-like background, restrained color used only for accents and interactive elements, serif for headlines, sans for body, monospace for small structural labels (eyebrows, tags, filenames). Never load external fonts or assets — everything must work offline in a single file.

Standard field set for the `<dl>`, in order: **Зачем** (the underlying problem it solves) → **Что делает** (the artifact shape it produces) → **Как вызвать** (the literal invocation, e.g. `/unknowns blindspot`, plus a one-line prompt phrasing) → **Пример** (one concrete, self-contained scenario — vary the domain across the set of breakdown cards on a page, e.g. mix client work, hiring, market research, event planning, so the reader sees the technique isn't tied to one field) → **Когда** (the one-line trigger condition).

**Slide deck** (keyboard-navigable full-screen presentation — use instead of a doc-style page when the artifact will be presented live or the user explicitly asks for a "deck"/"presentation"/"slides"):
```css
.slide { display:none; min-height:100vh; padding:80px 64px; flex-direction:column; justify-content:center; }
.slide.active { display:flex; }
.slide.title { align-items:center; text-align:center; background:var(--ink); color:var(--paper); }
.slide.title h1 { color:var(--paper); }
.slide-inner { max-width:920px; margin:0 auto; width:100%; }
.deck-nav { position:fixed; bottom:24px; left:50%; transform:translateX(-50%); display:flex; gap:8px; z-index:10; }
.deck-dot { width:8px; height:8px; border-radius:50%; background:var(--line-strong); border:none; cursor:pointer; padding:0; }
.deck-dot.active { background:var(--accent); width:22px; border-radius:4px; }
.deck-count { position:fixed; top:24px; right:28px; font-family:var(--mono); font-size:11px; color:var(--mute); }
```
```js
let slide = 0;
const slides = document.querySelectorAll('.slide');
const dots = document.querySelectorAll('.deck-dot');
function go(n) {
  slide = Math.max(0, Math.min(slides.length - 1, n));
  slides.forEach((s,i) => s.classList.toggle('active', i === slide));
  dots.forEach((d,i) => d.classList.toggle('active', i === slide));
  document.getElementById('deckCount').textContent = (slide+1) + ' / ' + slides.length;
}
document.addEventListener('keydown', e => {
  if (['ArrowRight','ArrowDown',' '].includes(e.key)) { e.preventDefault(); go(slide+1); }
  if (['ArrowLeft','ArrowUp'].includes(e.key))          { e.preventDefault(); go(slide-1); }
});
go(0);
```
Structure: one `<section class="slide" id="slideN">` per slide, first one `class="slide title"` as the cover; a `<div class="deck-nav">` with one `<button class="deck-dot" onclick="go(N)">` per slide plus `<span class="deck-count" id="deckCount">` — arrow keys / space to advance, click a dot to jump. Keep each slide to one idea; this is the format for **pitch** when it's meant to be presented rather than read async, and can substitute the hero+objections layout for that technique.

## Layout per technique (quick mapping)

- **blindspot / brainstorm / design-directions** → `.grid` of `.card`
- **teach-me** → single-column: eyebrow + prose ladder + one interactive worked example (native `<input type="range">`/`<select>`, styled minimally, no external libs)
- **mock** → full-bleed fake UI inside a `.card` framed like a device/window, clearly labeled "throwaway mock"
- **interview** → `.timeline-item` list of Q/A turns, ending in a table (`<table>`, plain, bordered with `var(--line)`) of decisions
- **reference-check** → two-column comparison (`display:grid; grid-template-columns:1fr 1fr`) of source vs. target excerpts
- **tweakable-plan** → ordered list of `.card` for tweakable decisions, then one `details.trusted` for the mechanical remainder
- **decision-notes** → `.timeline-item` list, one per deviation, each a `.card`
- **pitch** → hero block (outcome/demo first) → `.grid` of objection `.card`s → sign-off list
- **quiz** → report section, then a list of `.quiz-q`, each revealing correctness and a `.jump-back` anchor link to the relevant section id

Keep every page to one HTML file: inline `<style>` and `<script>`, no network requests, no build step.
